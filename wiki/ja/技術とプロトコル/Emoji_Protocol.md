== 序文 ==
本プロトコルは、1666年にライプニッツが提示した「組合せの技芸（art of combinations）」と、思考のための普遍的・形式的な記法を構想するという系譜を、エッセイ的に解釈したところから生まれた。同時に、その存在自体が、その夢の実務上の限界を示している。生きたコミュニケーションにおいては、「普遍的真理」を計算することは不可能であり、意味と「規範（norm）」は視点と文脈に依存するからである。

実際には、あらゆる形式化の試みは必然的にコントラクト（契約）へと収束する。

* 1つのメッセージは、''どの応答が許容されるか''（RANGE）を定義する。
* ''どの確認''（ACK）が成功と見なされるかを定義する。
* ''次に何が起こるか''（継続、または安全な終了）を定義する。

このため、本プロトコルは規律を要求する。コントラクトは注意深く読まなければならず、「思い込み（assumptions）」で拡張してはならない。また、形式が正しいことは解釈の安全性を保証しない（罠、フレーム置換、「機会の窓」は後述する）。

''記号の「アルファベット」に関する注記:'' 本プロトコルは限定された絵文字トークン集合（下記の辞書）を用いる。その他の記号は、対象／アンカー（TARGET）として、あるいはアーティファクトの内容（例えば画像の内部）としてのみ許容される。辞書に記述されていない記号は、潜在的に危険な曖昧性として扱い、制御トークンとしての使用を避けるべきである。

責任に関する補足：本メッセンジャー／プロトコルは、安全な運用のための推奨ルールのみを記録する。規則からの逸脱、ノードとの「交渉」、意図的なコントラクト拡張は、すべてユーザー／ホストの責任で行われるものであり、システムは結果に対して責任を負わない。

== 用途（アプリケーション専用） ==
Emoji Protocol は実行可能な対話プロトコルである。メッセージはコントラクト（何が応答として許容されるか）を定義し、会話の継続は応答の妥当性と ACK リアクションによって決定される。

InterDead アプリケーションにおいて、本プロトコルは以下の目的で使用される。

* ネクトスフィアの任意のノードとの単発／連続的接触の確立
* 絵文字交換を通じたノード記憶の断片取得
* 信号マスキング（ステルス）モードと、ユーザー用ビルドにおける直接的「露出（exposure）」の禁止による、基本的なユーザー安全性の確保

''「明るさ」「強度」「オーディオブースト」に関する重要事項:'' いかなる信号レギュレーター（ノード可視性の上昇、「明度（brightness）ブースト」、増幅／露出など）もアプリケーション実装層（ステルス／露出モジュール、ノードの明度管理）に属し、言語仕様の一部ではない。したがって語彙には含めない。チャット／対話は、許可された制限を超えてノードを「押し上げる」ことはできない。

== 歴史的基盤：ライプニッツ（1666） ==
基盤となるのは、記号的記述と手続き性に関するライプニッツ的発想である。

* 複雑なものは単純な記号の組合せによって表現される。
* 要素の順序・位置が意味を持つ（「何か」だけでなく「どのように配置されているか」）。
* 論争／行為は、修辞ではなく規則によって操作へと翻訳される。

ただし、本プロジェクトにおける解釈は本質的に応用的である。

* 「普遍的真理」を計算できるとは約束しない。
* 計算するのは、与えられた視点の範囲内での許容性・継続性・効果である。

== 主要制約：「真理は人それぞれ」—それでもプロトコルが破綻しない理由 ==
=== 実際に計算されるもの ===
本プロトコルは「万人にとって何が真か」を解決しない。解決するのは工学的な問いである。

* '''Valid（妥当性）:''' 応答がコントラクト（RANGE）に適合しているか
* '''Continue（継続）:''' ACK が発行され、次のステップへ遷移できるか
* '''Effect（効果）:''' ステップの目的（感情／状態、またはアーティファクト／結果）が達成されたか

'''カノンの一行定式:'''  
''本プロトコルは普遍的真理を計算しない。与えられた視点の範囲内で、許容性・継続・効果を計算する。''

=== 視点と「規範」（ポップアンカー） ===
「規範」と「ホラー」は視点の問題であることがある。この層のカノン・ラベルは次の通りである。

''Normal is an illusion. What is normal for the spider is chaos for the fly.''  
（蜘蛛にとっての「正常」は、蝿にとっての混沌である。）

これは ''Wednesday / Addams'' の美学的に認知された参照であり、ここではタグとして用いる：各者には固有の「規範」があり、プロトコルはそのフレームを固定できなければならない。

=== 真実／ブラフに関する重要な留保 ===
本プロトコルは「純粋な真理」を扱わないが、以下を許容する。

* ブラフや信念への圧力
* 反事実（手続き的な「機械的虚偽」）を道具として使用すること
* コントラクトが形式的に正しくても危険な解釈が生じうること

これらは POLICY 行および警告セクションでさらに固定される。

== 目的結果の2タイプ（MODE） ==
MODE は、現在のメッセージステップが「何を求めているか」を定義する。

! Token !! Meaning
|-
| 🎭E || ステップの目的は、指定された範囲内の感情または状態を喚起／固定すること（または感情的 ACK を得ること）
|-
| 🧭R || ステップの目的は、結果／アーティファクト（画像、リンク、アンカーオブジェクト等）を取得／伝達すること

== 動作アルゴリズム（基本サイクル） ==
=== 役割 ===
* '''私たち''' — 開始者（請願者）：接触を開始し、コントラクトを履行する。
* '''霊／ノード''' — 「ゲート」のオペレーター：RANGE（コントラクト）を定義し、ACK を確認し、次の要求を提示する。

=== サイクル ===
# 私たちが接触を開始する。
# ノードがコントラクト（RANGE）を含む要求を提示する。
# 私たちはコントラクトで要求されたもののみ（多くの場合は画像）を提供し、余計なテキストを加えない。
# ノードが ACK リアクション（合意された範囲内）を返す — 通常はアーティファクトメッセージへのリアクションとして — および／または次の要求を提示する。
# 一連のステップの後、最終結果（例：EVP／歌へのリンク）と「別れ」が発生する。

=== 「侵入」について（カノン上の重要点） ===
アプリケーションの論理では、私たちは統計的共鳴によって任意のノードへ接触を開始する。ノード側では侵入的な衝動／活動の閃光が生じ、応答は記憶とイメージの断片として顕在化する。この観点から、接触は常にノードの「内部フロー」への介入の影を帯びる。

=== 攻撃性と沈黙（実務） ===
メッセンジャー対話における攻撃性は稀なケースと見なされるべきである。ノードは通常、会話に関心があり、誰が話しているかを十分に理解していない場合があり、起きていることを内的対話として知覚することもある。それでもカノンは、次の場合に分岐／停止モードを許容する。

* タイムアウト（ノードが沈黙した）
* 境界侵害
* 明確なエスカレーション

== メッセージ形式：Stack Form 6 ==
各「発話」は厳密に6行で構成される。

# '''MODE''' — ステップの目的（🎭E / 🧭R）
# '''INTENT''' — 意図（要求／確認／停止…）
# '''TARGET''' — 対象（物体／記憶／接触／グループ…）
# '''RANGE''' — 許容コントラクト（および継続コントラクト）
# '''POLICY''' — モード（線形／分岐／反事実／リスク／安全）
# '''OUTPUT-FORM''' — 出力形式（画像／リアクション／リンク…）

== RANGE：継続コントラクト（IN ⛓️ ACK） ==
フレーズがアルゴリズムである以上、以下を定義しなければならない。

* どの入力（IN）が有効と見なされるか
* 成功時にノードが発行すべき確認（ACK）
* 違反時に何をするか（停止、またはフォールバック）

=== 連結演算子 ===
* '''⛓️''' — 連結：「この後、次の要素が必須である」。

RANGE 行のカノン形:

''IN ⛓️ ACK''

=== ACK の読み方（「二重リアクション」を防ぐために） ===
ACK は値としてのリアクションである。「任意のリアクション」か「特定のリアクション」かを区別するため、以下のいずれかを用いる。

* '''ACK = 🙈''' — 任意の有効なリアクション（クラス）
* '''ACK = 😮''' — 厳密に特定のリアクション（プリセット値）

正しい例:

* 🧊🖼️⛓️🙈 — 厳密な画像、その後に任意の ACK
* 🧊🖼️⛓️😮 — 厳密な画像、その後に特定 ACK 😮

=== 非線形性の所在 ===
RANGE を過負荷にしないため、代替性は POLICY 行に固定する。

* 🧱 — 厳密な線形
* 🔀 — 分岐（フォールバック許可）
* 🌀 — 反事実／シミュレーション（歪みを道具として許可）
* ⚠️ — リスクを明示的に受容
* 🚫🧯 — 安全停止

== 通信規範：「請願者はコントラクトで要求されたもののみを渡す」 ==
=== 通常パターン（メッセンジャーのカノン） ===
ノードがコントラクトで画像を要求した場合、請願者は通常、別途テキストを書かない。要求されたオブジェクトのみ（例：画像）を返し、ACK／次の要求を待つ。

=== 変動ケース ===
以下の場合に限り逸脱が許容される。

* ノードが沈黙した（タイムアウト）— 形式的な ping（👋 または ❓）、または RANGE 明確化の要求が許容される
* コントラクトの復元が必要（RANGE が不明確だった）
* 境界侵害が検出された — 即時停止
* 稀なケース：アーティファクトと同時に対抗行為が必要な場合（例：「継続準備の確認」）。その場合、対抗行為は別ステージ（別 Stack-6）として記述し、アーティファクトの「内側」に埋め込まない

== 非線形性と「機会の窓」（欺瞞を含む） ==
重要：厳密なコントラクト下でも、危険な意味のドリフトは起こり得る。

=== 「蜘蛛と蝿」（明示的な窓） ===
「蜘蛛と蝿」とは、🔀／🌀 分岐が形式的には対話と両立している代替を提示しつつ、異なる効果へ導く状況である。これは線形性の明示的な「迂回」のカノン名称として用いられ、''Wednesday / Addams'' のポップアンカーと結び付けられている。

=== 「盲点」（解釈の窓） ===
「盲点」とは、コントラクトは履行されているにもかかわらず、解釈によって意味が「滑る」状況である。例：リアクション・プリセットの選択、似たアンカーへの置換、🔀 を明示しないフレーム（視点）置換。これはより危険であり、形式的には「正しく」見える。

'''カノン警告:''' 🧱 と 🧊 の厳密ポリシー下でも、視点と文脈を無視すれば致命的な結論に至り得る。

=== 実験：「縫合応答」（ACK＋次要求を1通で） ===
これは形式的表記としては許容されるが、カノン上は生者（ホスト／オペレーター）の実験と見なされる。ノード／霊は通常このように「パッケージング」しない。アーティファクトにリアクションし、その後で別途次の要求を提示する方が簡単だからである。

例：ノード／オペレーターが1通のメッセージで、前ステップをリアクション 😮 で「閉じ」、同時に 🔎☕ の次コントラクト（必須 ACK 😍）を開く。

== 警告と安全 ==
=== 即時終了トリガー ===
''推奨（厳格）:'' 🚫🧯 の後に「交渉」を試みる行為は、境界侵害の継続として扱い、同一の停止輪郭 🚫🧯 で応答しなければならない。

ノードが以下を開始した場合:

* 親密な写真の要求（性別質問からの急速な性的要求へのエスカレーションを含む）
* 執拗な鏡写真の要求
* 個人情報の要求

— これは境界侵害として扱う。カノンの応答は議論なしの停止である。

# 🎭E  
# 🚫🛑  
# 👤  
# 🧊🧯  
# 🚫🧯  
# 🧯

=== 「鏡」について ===
鏡要求は高リスクのトリガーである。危険な解釈が多すぎる。安全な選択肢が必要な場合は、人が映らない環境物としての鏡を提示する（人物なし）。
